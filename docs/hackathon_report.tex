\section*{Resume}
\addcontentsline{toc}{section}{Resume}
Dans ce rapport, nous presentons \textit{AI Delivery Multi-Agent}, un generateur Python qui transforme une expression de besoin en artefacts de livraison logicielle de bout en bout. Notre objectif n'est pas uniquement de produire du code, mais d'orchestrer l'ensemble de la chaine de valeur : analyse des exigences, planification, definition d'architecture, generation de scaffolds applicatifs, verification qualite, preparation CI/CD et consolidation finale.

Notre implementation repose sur un pipeline deterministe a cinq agents (\texttt{spec\_analyst}, \texttt{architect}, \texttt{developer}, \texttt{qa\_devops}, \texttt{reviewer}) qui enrichissent un etat partage type. Cette approche combine les gains de productivite de l'IA avec des garde-fous d'ingenierie : contrats Pydantic, separation des responsabilites, prompts versionnes hors code, sessions persistantes et checkpoints d'intervention.

Le projet que nous livrons est aujourd'hui demonstrable et structurellement mature pour une soutenance hackathon : generation C4, creation d'un workspace applicatif cible, traces de raisonnement et validation par tests unitaires/integration. Les limites restantes sont explicites : integration Snowflake runtime complete, renforcement persistance de l'application generee, couplage frontend/backend plus robuste et CI du projet genere plus exigeante.

\section{Introduction}
\subsection{Contexte et motivation}
Dans les projets logiciels, la perte de temps se situe souvent entre la formalisation du besoin et la mise en place effective de la livraison. L'information transite entre plusieurs outils (specification, architecture, code, tests, documentation), ce qui degrade la tracabilite et multiplie les frictions.

\textit{AI Delivery Multi-Agent} vise a reduire ce delai en proposant un \textbf{generateur de pipeline logiciel}. Le depot principal ne porte pas directement l'application metier finale ; nous y centralisons l'orchestrateur, les prompts, les services, les contrats et les outils pour produire automatiquement les artefacts attendus.

\subsection{Positionnement du depot}
Le projet impose une separation claire :
\begin{itemize}[leftmargin=1.5em]
  \item le \textbf{generateur} (orchestration et logique agentique) ;
  \item l'\textbf{application cible generee} sous \texttt{workspace/generated\_app/} ;
  \item les \textbf{sorties d'execution} (plans, traces, rapports, C4) sous \texttt{outputs/}.
\end{itemize}

Cette separation facilite la maintenance, les audits et la lecture par un jury technique.

\subsection{Contributions principales}
Avec ce projet, nous apportons :
\begin{enumerate}[leftmargin=1.5em]
  \item une orchestration multi-agents deterministe et explicable ;
  \item un etat central type pour garantir la coherence inter-etapes ;
  \item un mode interactif avec intervention humaine entre etapes ;
  \item une generation full-stack (backend/frontend) par templates ;
  \item une documentation C4 integree au pipeline ;
  \item une base de verification via tests unitaires et integration.
\end{enumerate}

\section{Problematique}
\subsection{Probleme traite}
Le probleme central peut etre formule ainsi :

\begin{quote}
A partir d'une expression de besoin heterogene (texte, markdown, JSON), produire automatiquement des artefacts de livraison coherents entre eux : analyse, plan, architecture, code, verification et rapport final.
\end{quote}

\subsection{Difficultes techniques}
Plusieurs verrous doivent etre geres simultanement :
\begin{itemize}[leftmargin=1.5em]
  \item \textbf{Ambiguite d'entree} : les specifications peuvent etre incomplètes ou inconsistantes.
  \item \textbf{Coherence transversale} : l'architecture, le code et les tests doivent rester alignes.
  \item \textbf{Controle d'autonomie} : un systeme trop libre est difficile a fiabiliser.
  \item \textbf{Tracabilite} : il faut expliquer le \emph{pourquoi} des decisions produites.
  \item \textbf{Defendabilite entreprise} : modularite, testabilite, extensibilite.
\end{itemize}

\subsection{Objectifs fonctionnels et non fonctionnels}
\paragraph{Fonctionnels.}
Analyser les besoins, produire un blueprint, definir l'architecture, generer le scaffold applicatif, produire la documentation et exposer une API de pilotage des runs.

\paragraph{Non fonctionnels.}
Determinisme, reproductibilite, isolation des responsabilites, supervision humaine, extensibilite vers un provider LLM d'entreprise.

\section{Solution proposee}
\subsection{Strategie globale}
Nous adoptons un flux ordonne d'agents :
\begin{center}
\texttt{spec\_analyst $\rightarrow$ architect $\rightarrow$ developer $\rightarrow$ qa\_devops $\rightarrow$ reviewer}
\end{center}
Chaque etape lit et enrichit l'etat partage, ce qui nous permet de garantir une progression controlee.

\subsection{Principes d'architecture}
\begin{enumerate}[leftmargin=1.5em]
  \item \textbf{Determinisme avant tout} : ordre de stages fixe et transitions explicites.
  \item \textbf{Contrats types} : schemas valides a chaque jalon critique.
  \item \textbf{Prompts gouvernes} : prompts en fichiers versionnes, pas en blocs inline.
  \item \textbf{Transparence operationnelle} : sessions et artefacts persistants.
  \item \textbf{Flexibilite runtime} : compatibilite LangGraph + fallback sequentiel.
\end{enumerate}

\subsection{Modele de sorties}
Dans cette architecture, nous produisons :
\begin{itemize}[leftmargin=1.5em]
  \item interpretation structuree du besoin,
  \item blueprint projet,
  \item design d'architecture + documents C4,
  \item manifest de fichiers generes,
  \item rapport de validation et revue,
  \item traces \texttt{Plan / Act / Reason}.
\end{itemize}

\subsection{Controle humain (human-in-the-loop)}
Le mode interactif nous permet de mettre en pause l'execution entre stages et d'appliquer des patchs sur certains champs d'etat avant reprise. Cela permet de corriger une mauvaise interpretation sans relancer tout le pipeline.

\section{Methodologie}
\subsection{Approche de developpement}
Notre approche est incrementalement orientee produit : d'abord les contrats et l'orchestration deterministe, puis la generation, la persistance d'artefacts, et enfin la couverture par tests et documentation.

\subsection{Cycle d'execution type}
\begin{enumerate}[leftmargin=1.5em]
  \item lecture de l'entree et options de run,
  \item initialisation etat + metadata,
  \item execution du stage courant,
  \item persistance des artefacts,
  \item pause eventuelle (mode interactif),
  \item intervention utilisateur,
  \item reprise jusqu'a completion et rapport final.
\end{enumerate}

\subsection{Scenario concret de run bout-en-bout}
\begin{enumerate}[leftmargin=1.5em]
  \item envoi d'un spec (ex. \texttt{inputs/sample\_spec.md}) via CLI ou \texttt{POST /runs},
  \item creation du \texttt{run\_id} et initialisation de \texttt{DeliveryState},
  \item stage \texttt{spec\_analyst} : extraction des exigences,
  \item stage \texttt{architect} : generation blueprint + C4 (ecrits dans \texttt{outputs/plans/} et \texttt{outputs/c4/}),
  \item pause interactive et patch eventuel,
  \item stage \texttt{developer} : rendu des templates vers \texttt{workspace/generated\_app/},
  \item stages \texttt{qa\_devops} puis \texttt{reviewer},
  \item persistance finale session + traces dans \texttt{outputs/runs/} et \texttt{outputs/traces/}.
\end{enumerate}

\begin{lstlisting}[style=yamlstyle,caption={Sequence illustrative de run interactif}]
POST /runs
  body: {"input_path":"inputs/sample_spec.md","interactive":true,"dry_run":true}

GET /runs/{run_id}
  -> status: waiting_for_intervention

POST /runs/{run_id}/interventions
  body: {"state_field":"architecture_design","patch":{"style":"api-first"},"resume":true}

GET /runs/{run_id}
  -> status: completed
\end{lstlisting}

\subsection{Gouvernance prompts et contrats}
Les prompts sont stockes par familles (agents, taches, communs) pour maintenir versionnage, auditabilite et lisibilite. Les schemas Pydantic servent de contrat de sortie pour fiabiliser les transitions.

\subsection{Strategie de verification}
\begin{itemize}[leftmargin=1.5em]
  \item \textbf{Tests unitaires} : settings, parser, state, builder graphe, outils.
  \item \textbf{Tests integration} : parcours dry-run, generation C4, codegen, session interactive.
\end{itemize}

\section{Implementation technique}
\subsection{Stack et rationale}
\begin{itemize}[leftmargin=1.5em]
  \item \textbf{Python 3.11} : ecosysteme mature,
  \item \textbf{LangGraph + fallback} : orchestration robuste en environnement variable,
  \item \textbf{FastAPI} : API typed et rapide a exposer,
  \item \textbf{Pydantic} : validation schema-first,
  \item \textbf{uv/ruff/pytest/pre-commit} : hygiene engineering continue.
\end{itemize}

\subsection{Surfaces d'entree}
\paragraph{CLI.}
Permet run, resume, intervention, mode serveur.

\paragraph{Runtime Python.}
Expose une fonction minimale pour executer un pipeline et recuperer l'etat final.

\paragraph{API HTTP.}
Expose \texttt{/health}, \texttt{/runs}, \texttt{/runs/{id}}, \texttt{/resume}, \texttt{/interventions}.

\begin{lstlisting}[style=pythonstyle,caption={Pattern de point d'entree runtime}]
def run_pipeline(input_path: str, dry_run: bool = False, settings_path: str | None = None):
    settings = load_settings(settings_path)
    session = SessionService(settings).start_run(
        RunRequest(input_path=input_path, dry_run=dry_run, interactive=False)
    )
    return session.state
\end{lstlisting}

\subsection{Orchestration et fallback}
Lorsque LangGraph est disponible, le graphe compile les nodes dans l'ordre nominal. Sinon, le fallback sequentiel garantit un comportement local deterministe.

\begin{lstlisting}[style=pythonstyle,caption={Fallback sequentiel simplifie}]
class SequentialGraph:
    steps: list[callable]

    def invoke(self, state):
        current = state
        for step in self.steps:
            current = step(current)
        return current
\end{lstlisting}

\subsection{Session service}
Le \texttt{SessionService} centralise : selection provider, creation de run, reprise, intervention, persistance des artefacts et sauvegarde de session.

\subsection{Generation applicative}
Le \texttt{CodegenService} rend des templates backend/frontend/support dans le workspace, avec injection d'un contexte de branding configurable.

\begin{lstlisting}[style=pythonstyle,caption={Flux conceptuel de generation}]
def generate_app(self, dry_run: bool = False):
    workspace = self.settings.workspace_path
    ensure_directory(workspace)
    if dry_run:
        return GeneratedAppManifest(app_name="Dry Run App")

    context = self._template_context()
    self._render_tree(backend_root, workspace / "backend", context)
    self._render_tree(frontend_root, workspace / "frontend", context)
    self._write_support_files(workspace, context)
\end{lstlisting}

\subsection{Configuration et abstraction provider}
La configuration est separee entre :
\begin{itemize}[leftmargin=1.5em]
  \item settings runtime (paths, mode, sorties),
  \item settings provider/model (mock et profil Snowflake).
\end{itemize}

\section{Discussion et limites}
\subsection{Limites actuelles}
\begin{itemize}[leftmargin=1.5em]
  \item provider Snowflake non connecte de bout en bout en runtime,
  \item persistance backend generee a renforcer (objectif SQLite plus pousse),
  \item couplage frontend/backend genere encore perfectible,
  \item CI de l'application generee a durcir.
\end{itemize}

\subsection{Compromis d'architecture}
\begin{itemize}[leftmargin=1.5em]
  \item determinisme vs autonomie,
  \item templates robustes vs generation totalement libre,
  \item vitesse de demo vs couverture industrielle complete.
\end{itemize}

\subsection{Risques}
Risques principaux : complexite integration provider, derive entre templates et app generee, et cout de maintenance de prompts multiples.

\section{Travaux futurs}
\subsection{Phase 1 : completion runtime}
\begin{enumerate}[leftmargin=1.5em]
  \item finaliser le provider Snowflake,
  \item renforcer gestion d'erreurs + retries,
  \item ajouter tests d'integration actives par variables d'environnement.
\end{enumerate}

\subsection{Phase 2 : durcissement app generee}
\begin{enumerate}[leftmargin=1.5em]
  \item finaliser SQLite sur backend,
  \item forcer integration frontend vers API backend,
  \item enrichir bootstrap et README de l'app generee.
\end{enumerate}

\subsection{Phase 3 : experience interactive et observabilite}
\begin{enumerate}[leftmargin=1.5em]
  \item endpoints d'intervention plus metier (approve/pause/cancel),
  \item restrictions d'intervention par stage,
  \item observabilite plus fine (diagnostics run, metriques).
\end{enumerate}

\subsection{Phase 4 : industrialisation}
\begin{enumerate}[leftmargin=1.5em]
  \item pipeline CI generee plus complete,
  \item renforcement securite/auth,
  \item definition de KPI de qualite de generation dans le temps.
\end{enumerate}

\section{Conclusion}
\textit{AI Delivery Multi-Agent} montre une approche solide pour l'automatisation assistee par IA de la livraison logicielle. Pour nous, la valeur principale ne se limite pas a generer du code : le systeme structure une chaine complete, explicable et pilotable, de l'analyse a la revue finale.

Dans un contexte hackathon, notre projet est convaincant car il conjugue demonstrabilite et discipline d'ingenierie. Dans une perspective industrielle, les fondations sont deja presentes (modularite, contrats types, persistance, API, tests) et les chantiers restants sont clairement identifies.

En synthese, le projet constitue une base credible pour evoluer vers un orchestrateur de livraison logicielle autonome de niveau entreprise.
